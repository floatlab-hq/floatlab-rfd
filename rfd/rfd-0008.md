FL-RFD-0008: Full disk encryption with remote unlock features
===

**Author:** Chris Seufert & Gemini
**State:** `prediscussion`  
**Tags:** `storage`, `security`, `webauthn`

1. Introduction

This RFD proposes a mechanism for a zero-knowledge storage service that leverages the **WebAuthn PRF (Pseudo-Random
Function)** extension to derive encryption keys directly from a user’s passkey. Unlike traditional systems that rely on
user-managed master passwords, this design ties the encryption lifecycle to the hardware-backed security of a passkey,
ensuring that the service provider never has access to the user's data or keys.

2. Motivation

Traditional "zero-knowledge" storage often requires a master password, which is a significant point of failure for both
user experience (loss of password) and security (phishing or weak passwords). By using **Passkeys with PRF**, we can
provide:

- **Phishing-resistant authentication** that also unlocks data.
- **Hardware isolation** of the root encryption secret.
- **Lower friction** for users by using biometrics to perform both login and decryption in a single gesture.


3. Background

Current storage services typically encrypt data on the client using a key derived from a password via Argon2 or PBKDF2.
Passkeys (WebAuthn) traditionally only provide an **authentication** assertion. The **PRF extension** changes this by
allowing an authenticator (like a YubiKey or a phone's Secure Enclave) to output a consistent, secret 32-byte value
based on a salt provided by the server.

4. Proposed Design

The system will operate as a non-custodial storage service where the server functions as the **Relying Party (RP)**.

4.1. Key Derivation Flow

1. **Request:** During login, the server sends a challenge and a 32-byte `salt`.
2. **Authentication:** The client calls `navigator.credentials.get()` with the `prf` extension.
3. **Derivation:** The authenticator computes `HMAC-SHA256(SecretKey, Salt)`.
4. **Local Encryption:** The client uses this PRF output as the input to a Key Derivation Function (HKDF) to generate an
   **AES-256-GCM** key.

4.2. State Management

- **Server-Side:** Stores encrypted data blobs, the public key, and the credential ID.
- **Client-Side:** Performs all cryptographic operations in a secure context. No raw keys are ever sent to the server.


5. Technical Requirements

To support this architecture, the following environmental constraints must be met:

- **Hardware:** FIDO2 authenticators with PRF support (e.g., YubiKey 5 or **Apple Secure Enclave**).
- **Software:** Browsers supporting the `prf` extension (Chrome 108+, Safari 18+).
- **Standards:** Adherence to **WebAuthn L3** specifications.


6. Trade-offs and Considerations

6.1. Device Binding vs. Syncing

Using device-bound passkeys (hardware tokens) offers the highest security but risks data loss if the token is destroyed.
Synced passkeys (iCloud/Google) offer better recovery but slightly broader attack surfaces.

- **Recommendation:** Support both, but mandate a **Recovery Code** (BIP39 mnemonic) generated during setup.

6.2. Platform Inconsistency

As of mid-2025, Windows Hello supports passkeys but has inconsistent support for the PRF extension across all TPM
versions. The implementation must include a fallback or a clear "Hardware Incompatible" error state.

7. Security Impact

This design effectively eliminates "Master Password" phishing. Because the PRF secret is scoped to the **RP ID** (the
domain), it is impossible for a malicious site to trick the user into deriving the same key.
