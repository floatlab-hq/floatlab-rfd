FL-RFD-0007: Log Aggregation for Hyper-Constrained Edge Clusters
===


## 1. Introduction

We require a centralized logging solution for a very small (4G RAM) system or 2~3 node cluster. The target environment is characterized by severe memory pressure, with an ideal upper limit of **150MB RAM** per node for the logging service. The system must ingest both `systemd` (journald) logs and container logs (Docker).

## 2. Problem Statement

Modern observability stacks (ELK, SigNoz, Datadog) are built for "cloud-scale" environments where 2GB of RAM is considered "small." In a small system context—especially older models or those running multiple workloads—the memory overhead of indexing log content causes "Out of Memory" (OOM) events, leading to kernel panics or service instability.

### 2.1 Constraints

*   **Memory:** <150MB RSS (Resident Set Size) per node.
*   **Storage:** SSD longevity is a concern; minimize high-frequency random writes.
*   **Nodes:** 2~3-node distributed or single-point aggregation.

## 3. Comparison of Alternatives

| Metric | ZincSearch | OpenObserve | Grafana Loki |
| --- | --- | --- | --- |
| **Implementation** | Go  | Rust | Go  |
| **Indexing Strategy** | Full-text (Inverted Index) | Columnar (Parquet) | **Metadata Only** |
| **RAM Footprint** | ~300MB - 800MB | ~200MB - 500MB | **~80MB - 150MB** |
| **Search Speed** | Instant (Indexed) | Fast (Scanned) | Moderate (Grep-style) |
| **Architecture** | Monolithic | Monolithic/Distributed | Microservices/Single-Binary |

### 3.1 ZincSearch

ZincSearch provides a superior search experience but at the cost of memory-intensive indexing. Even with a small dataset, the heap requirements for managing inverted indexes make it unsuitable for a 150MB budget. Log search is not a primary feature, it is a troubleshooting tool.

### 3.2 OpenObserve

While OpenObserve is highly efficient (written in Rust) and offers a compelling "all-in-one" metrics/logs story, it struggles to stay under 150MB when ingesting multiple high-cardinality container logs. The overhead of the Parquet writer and the internal UI engine typically pushes the footprint beyond our target.

- - -

## 4. The Case for Loki (The Winner)

Loki wins this selection primarily through its **philosophical refusal to index log content.**

### 4.1 Architectural Advantages

1.  **Index Minimalization:** By only indexing labels (e.g., `container_name`, `host`), the index stays small enough to fit in the L2/L3 cache of a low-power ARM CPU.
2.  **Chunk-Based Storage:** Logs are compressed into chunks and flushed to disk. This is sequential I/O, which is much "healthier" for Raspberry Pi SD cards and consumer SSD, and also can take better advantage of compression than the random I/O required by traditional databases.
3.  **Decoupled Visualization:** Loki does not ship with a built-in UI. FloatLab itself will build a simple log search UI, using Loki as the backend.

### 4.2 Reaching the <150MB Target

To satisfy the constraint, Loki must be configured in "Single Binary" mode with the following adjustments:

*   **Disable Caching:** Remove `memcached` or in-memory chunk caches.
*   **Lower Flush Capacity:** Force chunks to disk earlier (at 1MB or 2MB) to prevent RAM accumulation.
*   **External Collector:** Use **Fluent Bit** (C-based) instead of Promtail (Go-based) to keep the ingestion agent under 20MB.

These containts may need to be adjusted over time, as we learn more about the resource usage patterns

- - -

## 5. Trade-offs and Risks

*   **Search Latency:** Searching for a specific string requires Loki to "grep" through compressed chunks. This could be slow for large ranges, however, generally we are only interested in very recent logs.
*   **Query Complexity:** Users must learn LogQL for advanced queries, FloatLab UI will present some basic filtering in the UI.
*   **Reliability:** At 150MB, there is little buffer for "log bursts." If a container enters a crash-loop and floods the logs, Loki may exceed the RAM limit.

- - -

## 6. Recommendation

Proceed with **Grafana Loki** as the aggregation point.

**Next Steps:**

1.  Deploy **Fluent Bit** to all nodes as a DaemonSet/Service.
2.  Deploy **Loki** to all nodes with `max_cache_freshness_per_query: 0`.
3.  Skip using the Grafana instance.

## 7. References

### 7.1 Example Loki Configuration
```yaml
auth_enabled: false

server:
  http_listen_port: 3100
  grpc_listen_port: 9096
  concurrent_request_limit: 5
  log_level: warn

common:
  instance_addr: 127.0.0.1
  path_prefix: /floatlab/system/loki_data
  storage:
    filesystem:
      chunks_directory: /floatlab/system/loki_data/chunks
      rules_directory: /floatlab/system/loki_data/rules
  replication_factor: 1
  ring:
    kvstore:
      store: inmemory

ingester:
  # Internal compression: LZ4 is a good trade-off between compression and speed.
  # Investigate Zstd, hwoever need to check ram usage
  chunk_encoding: LZ4
  # RAM Management: Flush small 512KB chunks to disk frequently
  chunk_target_size: 524288
  max_chunk_age: 15m
  chunk_idle_period: 5m
  wal:
    enabled: true
    dir: /floatlab/system/loki_data/wal

# 21-Day Retention Logic
limits_config:
  retention_period: 504h
  ingestion_rate_mb: 2
  ingestion_burst_size_mb: 4
  max_query_parallelism: 1

compactor:
  working_directory: /floatlab/system/loki_data/retention
  shared_store: filesystem
  compaction_interval: 10m
  retention_enabled: true
  retention_delete_delay: 2h
  retention_delete_worker_count: 50

schema_config:
  configs:
    - from: 2024-01-01
      store: tsdb
      object_store: filesystem
      schema: v13
      index:
        prefix: index_
        period: 24h

query_range:
  cache_results: false # Essential to stay under 150MB RAM

```

### 7.2 ZFS Configuration
```sh
# Set recordsize to 1M to match Loki's chunk flushes
zfs set recordsize=1M floatlab/loki

# Use LZ4: It's nearly 'free' CPU-wise and catches what Loki might miss
zfs set compression=lz4 floatlab/loki

# Disable access time updates to save metadata writes on your SSD/HDD
zfs set atime=off floatlab/loki
```
