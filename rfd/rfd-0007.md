FL-RFD-0007: Log Aggregation for Hyper-Constrained Edge Clusters
===


## 1. Introduction

We require a centralized logging solution for a very small (4G RAM) system or 2–3 node cluster. The target environment is characterized by severe memory pressure, with an ideal upper limit of **150MB RSS** per node for the logging service. The system must ingest both `systemd` (journald) logs and container logs (Docker).

## 2. Problem Statement

Most observability platforms (Elasticsearch-based stacks, Datadog-class agents, and many all-in-one suites) are tuned for environments where multi-GB memory overhead is acceptable. In our constrained edge setup, heavy indexing and large in-memory metadata structures increase OOM risk and reduce system stability.

### 2.1 Constraints

*   **Memory:** <150MB RSS (Resident Set Size) per node for the log backend.
*   **Storage:** SSD longevity is a concern; prefer sequential writes and high compression.
*   **Nodes:** 2–3-node distributed or single-point aggregation.
*   **Data Shape:** Must tolerate high-cardinality fields (e.g., `user_id`, `trace_id`, session IDs, container hashes).

## 3. Comparison of Alternatives

| Metric | Elasticsearch | Grafana Loki | VictoriaLogs |
| --- | --- | --- | --- |
| **Operational Complexity** | High (cluster, shard, JVM tuning) | Medium (labels, topology choices) | **Low (single binary, zero-admin)** |
| **Indexing Model** | Full-text inverted index | Label index + chunk scans | **Optimized log indexing without schema/mapping setup** |
| **Resource Efficiency** | Highest overhead | Moderate | **Best-in-class disk + RAM efficiency** |
| **High Cardinality Handling** | Costly at scale | Risky when labels explode | **Designed to handle high-cardinality fields** |
| **Query UX** | Lucene/KQL complexity | LogQL | **LogsQL (human-friendly SQL/pipe style)** |

### 3.1 Elasticsearch

Elasticsearch remains powerful but is not aligned with our constraints. It requires active management of JVM heaps, shard strategy, and cluster health. The operational and memory overhead is disproportionate for a 150MB/node target.

### 3.2 Grafana Loki

Loki is significantly lighter than Elasticsearch, but high-cardinality label patterns can still become a scaling hazard. Our workload contains fields we want to query directly (user, session, trace, and infra identifiers), making strict cardinality discipline a long-term operational burden.

## 4. The Case for VictoriaLogs (Primary Choice)

VictoriaLogs is selected because it combines low operational friction with excellent efficiency and modern querying ergonomics.

### 4.1 Zero-Admin Simplicity

1.  **Single Dependency-Free Binary:** No JVM, no shard planning, and no complex multi-service control plane required for small deployments.
2.  **"Just Works" Ingestion:** No upfront mappings/schemas required; logs can be sent directly and indexed automatically.
3.  **Local-Friendly by Design:** Lightweight enough for very small edge nodes while still scaling to large production datasets.

### 4.2 Radical Resource Efficiency

1.  **Disk Reduction:** Real-world reports and benchmark narratives consistently position VictoriaLogs as dramatically smaller on disk than Elasticsearch (often ~10x–15x), driven by compact storage techniques (columnar layout + delta encoding + strong compression).
2.  **Low Memory Pressure:** Uses compact structures (including Bloom-filter-based data skipping) instead of keeping a massive full-text index resident in memory, enabling much lower RAM usage (often cited as up to an order-of-magnitude+ better than Elasticsearch-class systems).
3.  **Edge Suitability:** The efficiency profile directly supports our <150MB operating objective and SSD-write concerns.

### 4.3 High-Cardinality Superpowers

Many logging backends treat high-cardinality metadata as an anti-pattern that can degrade performance or destabilize clusters. VictoriaLogs is built to tolerate this data shape well, enabling practical indexing and querying across fields like:

*   `user_id`
*   `trace_id`
*   `session_id`
*   IP addresses
*   container hashes

This is a strong fit for microservices troubleshooting and security/audit workflows.

### 4.4 LogsQL: Querying Built for Humans

LogsQL offers an approachable syntax combining filter-first search with pipeline-style analytics.

*   **Simple search:** `_stream:{app="nginx"} error`
*   **Fast aggregation:** `_stream:{app="nginx"} error | stats count() by status_code`

This lowers the operational learning curve compared with traditional Lucene-like query models.

### 4.5 Seamless Modern Integration

VictoriaLogs integrates cleanly with existing tooling:

*   **Grafana plugin** for dashboards and shared visualization workflows.
*   **Standard collectors** including Fluent Bit, Logstash, Vector, and direct HTTP ingestion (e.g., `curl`).

## 5. Trade-offs and Risks

*   **Ecosystem Familiarity:** Team members may be more familiar with Elasticsearch/Loki patterns; some onboarding is expected.
*   **Query Language Adoption:** LogsQL is simpler than many alternatives, but still requires basic team enablement.
*   **Sizing Discipline:** Even with strong efficiency, bursty workloads still require retention/rate controls to stay within strict edge budgets.

- - -

## 6. Recommendation

Proceed with **VictoriaLogs** as the primary log aggregation backend for FloatLab edge clusters.

**Next Steps:**

1.  Deploy **Fluent Bit** to all nodes as a DaemonSet/Service and route logs to VictoriaLogs.
2.  Deploy **VictoriaLogs** in single-binary mode with conservative retention/rate limits aligned to <150MB RSS targets.
3.  Keep **Grafana** optional, enabled only when dashboards are required.
4.  Publish a short internal query cookbook with core LogsQL patterns for operations and incident response.
