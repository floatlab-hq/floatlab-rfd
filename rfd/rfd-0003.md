# Workload Hosting

### `FL-RFD-0003`

FloatLab will host workloads in Containers, using the open-source Docker tooling. FloatLab will expose each workload via
the web admin interface as a `docker-compose.yml` file that can be deployed to a specific host, and fail-over to a
nominated host.

FloatLab will not be operating as a cluster, see [RFD-0004](./rfd-0004.md) for more details

FloatLab will assist the user in creating, and editing `docker-compose.yml` files, by both providing an editor that
can validate the file, assist in configuring volumes, editing ENV variables, injecting sidecars and assigning networks. FloatLab will store
FloatLab-specific deployment metadata directly in the compose file using `x-fl-*` extension fields, including:

- Snapshot schedule
- Primary host
- Secondary host, replication schedule, and failover priority
- Backup destination and schedule

**Reasons for basic workload definitions in `docker-compose.yml`:**

- Single configuration file for each workload
- Very common for projects to publish example `docker-compose.yml` saving the user from having to translate the config
- Contains enough configuration to deploy nearly any workload
- Does not contain complicated features like auto-scaling, auto-placement, or other complex orchestration logic
- Allows for multiple containers in a single "stack"
**Reasons against using `docker-compose.yml`:**

- `docker-compose.yml` is not a standard format, and is not portable between different orchestrators
- `docker-compose.yml` has no support for adding additional configuration required by FloatLab

We see this as a fair trade-off compared to using docker swarm, kubernetes, or helm-charts for workload deployment. All of which require unique definitions and are less portable than a standard docker-compose.yml that can work on any machine running docker.

### Deployment & Control Plane Options

The following options were considered for workload hosting and operations control:

#### Option A: Docker Engine + `docker-compose.yml` + FloatLab `x-fl-*` extension metadata (**Recommended**)

- Workload definition remains user-facing and portable via `docker-compose.yml`
- FloatLab-specific behavior is captured in explicit `x-fl-*` extension fields in the same file
- Suitable for single-host and warm-standby/failover patterns without introducing cluster complexity
- Lowest operational overhead for target user profile and current product phase

#### Option B: Docker Swarm

- Adds built-in scheduler and service abstraction
- Requires translating user expectations from plain compose behavior into swarm semantics
- Introduces additional control-plane operations burden with limited alignment to current requirements

#### Option C: Kubernetes (direct manifests or Helm)

- Highest flexibility and ecosystem support
- Substantially higher complexity in platform operation, packaging, and user support
- Poor fit for current goals focused on straightforward, portable, non-cluster deployments

#### Option D: Separate companion file for FloatLab metadata

- Separates metadata from workload definition but introduces dual-file operational complexity
- Increases drift risk between runtime config and policy config
- Adds import/export and lifecycle overhead compared to a single compose artifact

#### Decision Criteria

- User portability of workload definitions
- Operational simplicity for both user and FloatLab
- Clear, predictable namespacing of platform policy config inside compose (`x-fl-*`)
- Ability to support backup/replication/failover in non-cluster mode
- Future migration path to more advanced orchestration if required

### Capabilities Required

The following capabilities are required to make this architecture operable.

#### Authoring & Validation

- Compose-aware editor with schema validation and linting
- Validation of `x-fl-*` extension schema and required metadata presence
- Validation of unsupported/unsafe fields based on FloatLab policy
- Assisted editing for ENV variables, volumes, sidecars, and network assignment
- Deterministic rendering and diff preview before deployment

#### Deployment Orchestration

- Host inventory and host eligibility checks (runtime version, storage/network prerequisites)
- Idempotent deployment actions (`create`, `update`, `rollback`)
- Health-gated rollout with timeout and failure handling
- Deployment state tracking per workload and per host

#### Failover & Replication

- Primary/secondary host assignment and failover priority
- Replication scheduling aligned with volume snapshot policy
- Explicit failover runbook logic (automatic triggers and manual override)
- Failback support with data consistency checks

#### Data Protection

- Snapshot policy definition and execution scheduling
- Backup destination management (local/remote targets) and retention policy
- Restore workflows with pre-flight validation and post-restore verification
- Backup integrity checks and periodic restore drills

#### Secrets Management

- Secret creation/import/update with at-rest encryption
- Scoped secret injection into workloads at deploy/runtime
- Secret rotation workflow with minimal workload disruption
- Audit trail for secret access and mutation events

#### Storage & Volume Management

- Automatic volume/sub-volume provisioning per workload/container mount
- Policy controls: quota, block size, compression, and mount options
- Path translation and validation when compose files are edited manually
- Volume migration/reattachment support during host failover and recovery

#### Networking

- Per-container IP allocation with conflict detection
- Service discovery and stable naming between containers in a workload
- Ingress/egress policy controls and optional host-port exposure
- Network policy validation during compose/deploy workflow

#### Observability & Operations

- Centralized logs, metrics, and events for workloads and platform actions
- Health status model (container, workload, host, replication, backup)
- Alerting hooks for failed deployments, replication lag, and backup failures
- Operator actions audit log and timeline view

#### Security & Governance

- RBAC for workload administration and operational actions
- Image provenance and vulnerability policy checks before deployment
- Least-privilege defaults for container runtime settings
- Change approval controls for high-risk operations (failover, restore, policy change)

#### Lifecycle & Compatibility

- Versioned `x-fl-*` metadata schema with migration tooling
- Backward compatibility handling for prior workload definitions
- Import/export packaging for workload portability between FloatLab instances
- Documented deprecation policy for config fields and behaviors

### Minimum Delivery Scope (MVP)

The minimum scope required for initial release:

- Compose authoring/validation with policy checks
- Deployment to primary host with tracked state
- Secondary host designation with manual failover
- Snapshot + backup scheduling with restore capability
- Basic secrets injection and encrypted storage
- Core observability (status, logs, critical alerts)

### Post-MVP Hardening

- Automatic failover with configurable safeguards
- Secret rotation workflows and enhanced audit controls
- Backup integrity automation and regular restore validation
- Expanded governance controls (approvals, policy packs)
- Compatibility tooling for schema evolution across releases

### Secrets Management

Each workload will allow for the storage of secrets with-in the workload volume. These can be imported into the
workload using the `docker-compose.yml` file.

### Sidecar Configuration Alternative

FloatLab-specific operational metadata should be represented using `x-fl-*` extension fields in `docker-compose.yml`,
rather than overloading container labels intended for runtime/container discovery concerns.

### Volume Management

Float lab will automatically create a volume for each workload, and a sub-volume for each container volume mount. These
can have additional settings (eg Disk Quote, Block Size, Compression). These volume paths will be automatically
translated into the correct paths, and validated after manual file editing.

### Network Management
Each container will automatically receive a unique IP address rather than the default of sharing the host's IP. This will reduce the contention for port numbers, reduce the need for reverse proxies and help simplify the migration of containers to different hosts.
